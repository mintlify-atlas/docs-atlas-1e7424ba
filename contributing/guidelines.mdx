These guidelines exist to keep the Open Skills catalog high-quality, authentic, and respectful to both the personas being distilled and the people who use them. Reading them before you submit will save you time and help your pull request get merged faster.

## Quality standards

A good skill produces responses that feel genuinely like the persona — not a caricature or a generic AI voice with a name attached.

- Skills should faithfully represent the persona's authentic thinking style, not just their most famous quotes or surface-level traits
- Include sample interactions that demonstrate the skill working well across different types of questions
- Write a clear, descriptive README in your skill repository so reviewers and users understand what the skill does
- Test your skill before submitting — load it into an AI tool, run several conversations, and check whether the responses feel authentic

## Content guidelines

Skills of real people require care. The goal is to capture how someone thinks and communicates, not to fabricate positions or statements they never expressed.

- Skills of real living people should focus on their public persona, published ideas, interviews, and documented communication style
- Don't put words in people's mouths beyond what their documented thinking patterns support — extrapolating is fine, inventing is not
- Skills of deceased figures or fictional personas have more latitude for creative interpretation
- No harmful, deceptive, or malicious persona definitions — this includes skills designed to impersonate someone for fraud, spread disinformation, or cause harm

## PR checklist

Before you open a pull request, confirm each of the following:

<AccordionGroup>
  <Accordion title="Repository requirements">
    - [ ] GitHub repository is public
    - [ ] Repository has a README explaining the skill
    - [ ] Skill name follows the format: `handle/skillname-skill`
  </Accordion>

  <Accordion title="PR requirements">
    - [ ] PR adds exactly one row to the correct category table in `README.md`
    - [ ] The same row is added to `README_EN.md` (English version)
    - [ ] Row format: `| SkillName.skill | Description | [handle/repo](url) |`
    - [ ] Description is concise (under 50 characters ideally)
  </Accordion>
</AccordionGroup>

## Community values

Open Skills is built on the philosophy that **everything can be a skill, life is full of distillation**. The most interesting contributions are often the most unexpected ones.

Don't limit yourself to famous people or historical figures. The catalog already includes colleagues, exes, parents, a Buddhist master, a retail investor, and an ancient matchmaker. Roles, relationships, moments, abstract concepts — if it has a recognizable cognitive or emotional pattern, it can be a skill.

The spirit of the project is creative and generous: distill the people and ideas around you, share them with the community, and let others build on what you've made.
